JSON Web Tokens (JWTs) are JSON formatted strings which contain information about a subject. The subject is typically a user, but is the identity for any system that needs to prove itself to other systems, like when an API calls another API during machine-to-machine communication. The JWT is generated after the subject authenticates with the system by entering their username/password, or authenticating any other way.

The IETF's [RFC7519](https://datatracker.ietf.org/doc/html/rfc7519) is the official standard for JWTs.

## Reading JWTs

JWTs are formatted using the JSON Web Structure (JWS) format defined in [RFC7515](https://datatracker.ietf.org/doc/html/rfc7515). This means they are formatted into three sections (Header, Payload, Signature), separated by a dot. Each section is a self-contained, Base64 URL Encoded, JSON value.

> Note: Encoded does not mean encrypted. Anyone can decode a JWT and read the plaintext values. The JWT is Base64 URL Encoded to make transmitting the string between services simple by avoiding characters that can be problematic on the web, like spaces and quotes.

For an example, the JWT value `eyAiaGVhZGVyIiA6ICJ0aGUtaGVhZGVyIiB9LnsgInBheWxvYWQiIDogInRoZS1wYXlsb2FkIiB9LnsgInNpZ25hdHVyZSIgOiAidGhlLXNpZ25hdHVyZSIgfQ==` translates to `{ "header": "the-header"}.{"payload":"the-payload"}.{"signature": "the-signature"}`.

Many libraries exist to parse a JWT in a variety of programming languages. If you want to make a JWT string human readable, you can use a JWT Decoder like the one at [Duende's demo site](https://demo.duendesoftware.com/jwt-decoder).

### Generating a JWT with code

If it helps to visualize the process of generating a JWT in terms of code, the below pseudocode describes shows what steps are taken to generate one:

```
//Create the Header and Payload JSON strings
var header = "{ "alg": "HS256", "typ": "JWT" }";
var payload = "{"sub":"user@email.com"}";

//Base64 URL Encode the Header and Payload values
var headerBase64 = Base64Encode(header);
var payloadBase64 = Base64Encode(payload);

//Generate the signature
// Create the value, sign it with the algorithm mentioned in the header, then Base64 URL Encode it
var signature = $"{headerBase64}.{payloadBase64}";
var signingSecret = LoadSigningSecretFromSomewhere();
var signedSignature = SignWithHmacSha256(signature, signingSecret);
var signedSignatureBase64 = Base64Encode(signedSignature);

var jwt = $"{headerBase64}.{payloadBase64}.{signedSignatureBase64}";
print(jwt);
```

### Header

The header, also called the JSON Object Signing and Encryption Header (JOSE Header), contains properties describing what cryptographic operations were performed to sign the Payload of the JWT. The example below shows that the HS256 algorithm was used to sign the Payload.

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

### Payload

The payload section of the JWT contains the Claims for the subject, which is information about the identity. This includes custom claims that are specific to your applications, and a number of Registered Claim Names, a set of claims with known meanings.

#### Registered Claim Names

Some examples of Registered Claim Names are:
- `sub`: The subject the JWT is for. This can be any string that uniquely identifies the subject, like an email address or GUID.
- `iat`: The date/time the JWT was issued. The value is the number of seconds since January 1, 1970. 
- `exp`: The date/time the JWT expires. Systems receiving a JWT should not trust one after its expiration time has passed. The value is the number of seconds since January 1, 1970. 

The full set of Registered Claim Names are listed in the [JWT RFC](https://datatracker.ietf.org/doc/html/rfc7519#section-4.1).

#### Custom Claims

An identity system can add any number of claims it wants. Some systems will add everything it knows about the subject, others will scope them only to the system that will use the JWT.

#### Real World Example

The below example shows a payload with a mix of claims. The `sub` and `exp` are registered claims, and the rest are custom claims used by the consuming application.

```json
{
  "sub": "user@example.com",
  "exp": 1300819380,
  "client_id": "my-frontend",
  "": "",
  "scope": [
    "admin"
    "orders:read",
    "inventory:read"
    "inventory:write"
  ]
}
```

> Note: A JWT may contain every claim for a user, making the transmitted string very large. Conversely it can contain a minimum amount of information about the subject. The amount of information contained in the JWT is determined by the service generating it.

### Signature

The signature is used to verify the JWT has not been tampered with since it was created. Because the token is an arbitrary string, the Signature adds trust that the JWT is the same value that it was when it was generated by the source of truth.
